Architecture proposal

Core CLI + GUI

把 SealMail 从「桌面应用里的 Rust 后端」升级为「CLI 是唯一业务入口，GUI 是展示壳」的结构。Core 承载业务规则，CLI 暴露稳定 JSON 合约，GUI 只通过 CLI bridge 读取数据和发出用户意图。

1
业务只进 Core 收发信、同步、签名、验证、过滤、缓存、草稿、联系人、密钥状态都由 Core 统一定义。
2
业务入口只走 CLI GUI 不直接调用 Core。Tauri 只负责启动 CLI、传入 JSON/stdin/env、读取 JSON stdout。
3
发布物先保持简单 用户仍安装一个签名公证过的桌面 App；GUI 可以启动内置 CLI，或以 App 自身的 CLI 模式运行。

目标形态

GUI 不再和 CLI 并列消费 Core；GUI 通过 CLI JSON 合约消费业务能力，CLI 再调用 Core。

关系组成

每层只负责一件事。越靠下越稳定，越靠上越贴近用户体验。

Layer 01
Domain Model 账户、邮件、目录、身份、签名、可信联系人、规则、草稿、错误和事件的统一语义。
Layer 02
Core Services 把业务用例做成稳定能力：同步、发送、验证、过滤、缓存读写、风险判断。
Layer 03
Adapters 连接外部世界：IMAP、POP3、SMTP、OAuth、Ledger、SQLite、系统通知、文件系统。
Layer 04
Interface Hosts CLI 是 Core 的稳定产品接口；Tauri command 只保留 CLI bridge 和系统能力。
Layer 05
Presentation React GUI 只展示状态、收集用户意图、订阅事件，不拥有业务规则。

接口面

接口按「意图」分组，而不是按屏幕分组。GUI 调用 CLI 的 JSON 形态，CLI 调用 Core。

Core 对外能力

  • 账户与身份添加、测试、移除账户；选择本地身份或硬件身份；读取可信状态。
  • 邮件与目录列目录、同步、分页读取、读完整邮件、移动、归档、删除、标记。
  • 发送与草稿保存草稿、删除草稿、发送邮件、附加签名、处理附件。
  • 信任与风险验证签名、识别冒充、添加可信联系人、输出风险拦截原因。
  • 规则与自动化管理过滤器、应用过滤器、收集联系人、后台任务状态。

入口接口职责

  • GUI Adapter把用户意图转成 CLI 参数/stdin，把 CLI JSON 转成视图模型；同时处理窗口、文件选择、外部链接、通知点击。
  • CLI Adapter把命令参数转成 Core 意图，输出人类可读文本或机器可读 JSON。
  • Daemon API提供本机 IPC 或 localhost API，维护长连接事件流和后台同步队列。
  • Release Adapter处理 App 更新、版本检查、签名包验证、内置 sidecar 生命周期。
  • Test Harness直接驱动 Core 用例，验证业务结果，不依赖 GUI 或真实点击。

运行时态

同一套 Core 由 CLI 加载。GUI 通过一次性 CLI 进程完成命令和查询，后续可演进为 daemon。

GUI + CLI bridge 模式
启动Tauri App 只启动 GUI 和平台 adapter。
交互React 调用 cli_json，Tauri 启动 sealmail-cli 或 App CLI 模式。
输出CLI 读取统一配置目录，调用 Core，并把 JSON stdout 返回 GUI。
CLI 一次性模式
启动CLI 加载同一个 Core 配置目录和本地缓存。
执行完成同步、查询、发信、诊断等单个意图。
输出输出文本或 JSON，进程结束，不保留 UI 状态。
Daemon 常驻模式
启动后台进程持有 Core，维护同步计划和事件队列。
服务GUI 与 CLI 都连接 daemon，请求快照或订阅事件。
更新跟随签名 App 更新；重启前保存任务和状态。
推荐演进顺序： 第一版采用方案 A：GUI 通过 CLI JSON bridge 调用业务能力。这样所有用户向能力都能被 CLI、CI 和真实邮箱 E2E 直接测试。等后台同步和事件流需求稳定后，再把 CLI 背后的执行体替换为 daemon。

通信方式

通信分为命令、查询、事件三类。命令改变世界，查询读取快照，事件通知变化。

命令 Command

发送邮件、移动邮件、添加账户、信任联系人、应用过滤器。要求明确成功或失败，并返回业务结果。

查询 Query

获取账户列表、读取邮件列表、打开邮件详情、读取草稿、读取可信状态。要求可分页、可缓存、可离线。

事件 Event

同步进度、新邮件、验证状态变化、过滤器执行完成、通知点击。GUI 订阅事件并更新展示。

GUI 通信

GUI 只调用 Tauri 的 cli_json bridge。bridge 启动 CLI 进程，传入参数、stdin 和必要 env，解析 JSON stdout；平台能力如 OAuth、通知、外链仍由 Tauri adapter 处理。

CLI 通信

一次性命令直接调用 Core；daemon 存在时优先连接 daemon，避免重复同步和状态竞争。

错误通信

错误要携带类别、动作、对象和用户可理解说明；不要用默认值掩盖本不该为空的状态。

发布与更新边界

架构可以拆，但发布物先保持集中。自动更新、签名和公证围绕最终 App bundle 管理。

推荐发布形态

  • 桌面 App继续作为主安装入口，包含 GUI、CLI 模式和 Core。
  • 内置 CLI优先使用 App 自身的 CLI 模式，避免额外 sidecar 签名负担；开发环境也可使用独立 sealmail-cli。
  • 独立 CLI作为诊断和自动化入口随源码构建，后续可作为独立发行物单独签名、公证。
  • 更新策略优先更新整个签名 App bundle，避免运行时替换未签名二进制。

安全边界

  • 密钥材料只在统一配置目录和系统权限内流转，不进入仓库，不作为日志输出。
  • OAuth tokenCore 统一刷新和保存，GUI 不直接拥有 token 处理逻辑。
  • Ledger硬件交互通过设备 adapter 触发，签名确认仍由用户在设备上完成。
  • 更新签名更新包必须经过 updater 签名验证，App bundle 走平台签名和公证。

迁移路线

每一步都应该能独立合并和验证，避免一次性重写。

Phase 1

定义 Core 边界

梳理现有 Tauri command，把业务用例按账户、邮件、信任、规则、状态分组，确定统一 DTO 和错误模型。

Phase 2

抽离业务服务

把业务逻辑从 Tauri 层移到 Core，并让 CLI 成为 Core 的稳定 JSON 入口。

Phase 3

GUI 改走 CLI bridge

GUI 业务 API 不再直接 invoke Core command，而是调用 cli_json bridge；bridge 启动 CLI 或 App CLI 模式并解析 JSON。

Phase 4

引入 Daemon

把长期同步、通知和事件订阅迁到后台服务。GUI 逐步变成状态展示和交互客户端。

SealMail Core CLI + GUI 架构文档。关注组成关系、接口边界、运行时态和通信方式；实现细节应沉淀到后续 ADR 与任务拆分中。
