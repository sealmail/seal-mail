# 面向产品哲学的 AI 测试方法论

我们想解决的问题，不是“让 AI 多写几个测试”。

真正的问题是：如何让 AI coding 不只是实现功能，而是持续逼近一个真正好的产品。

如果产品哲学是目标，那么 AI 测试方法论就是让 AI coding 不偏航的导航系统。

对于 SealMail 来说，这个目标已经很清楚：

> 它首先必须是一个好用的邮件客户端。  
> 收、看、写、发、同步、附件、草稿、归档、删除这些基本体验必须顺手可靠。  
> 安全、签名、验证是特色能力，但它们应该增强邮件体验，而不是压过邮件体验。

所以我们的测试方法论，也不能只围绕“特色功能是否工作”展开。

它必须回答一个更重要的问题：

> AI 写出来的东西，是否让产品更接近用户愿意每天使用的状态？

## 一、测试不是找 bug，而是保护产品方向

很多时候，我们把测试理解成“检查代码有没有错”。

这个理解太窄。

测试当然要找 bug，但对一个产品来说，更重要的是保护方向。

一个功能可以没有明显 bug，但仍然让产品变差。

比如：

- 发信功能能跑通，但写信流程变复杂了。
- 安全提示准确，但把邮件列表变得很吵。
- 同步逻辑正确，但用户不知道当前看到的是缓存还是最新邮件。
- 架构更优雅了，但 GUI 的日常操作变慢了。
- 测试全绿，但用户第一次添加账户还是很困惑。

这些都不是传统单元测试容易发现的问题。

所以 AI 测试不能只问：

> 代码对不对？

还要问：

> 这个改动有没有伤害产品的基本体验？  
> 它有没有让特色功能喧宾夺主？  
> 它有没有增加用户的认知负担？  
> 它有没有让异常场景更诚实、更可恢复？

测试的本质，是把产品哲学变成可反复检查的证据。

## 二、先定义产品基本盘，再定义特色能力

AI coding 很容易被“显眼的功能”吸引。

因为特色功能更容易描述，也更容易让实现看起来有成果。

但用户是否愿意日常使用一个产品，首先取决于基本盘。

所以任何产品的 AI 测试，都应该先分两层：

### 1. 基本盘测试

基本盘测试回答：

> 这个产品作为它所属品类，是否已经足够好用？

对 SealMail 来说，就是：

- 能否稳定添加账户？
- 能否快速看到邮件列表？
- 能否顺利打开邮件？
- 能否自然写信、保存草稿、发送邮件？
- 能否处理附件？
- 能否移动、归档、删除、标记邮件？
- 能否退出重开后恢复状态？
- 网络失败、密码错误、服务器异常时，用户是否知道发生了什么？

这些不是“普通功能”，而是产品的生命线。

如果基本盘不顺，特色功能越强，反而越像负担。

### 2. 特色能力测试

特色能力测试回答：

> 这个产品和同类产品相比，独特价值是否真实存在？

对 SealMail 来说，就是：

- 签名是否真的可验证？
- 篡改是否能识别？
- 冒充是否能拦截？
- 可信联系人是否能帮助用户判断？
- 高风险邮件是否能给出清楚、克制、可行动的提示？

特色能力必须被测试，但它不能排在基本盘前面。

更准确地说：

> 基本盘决定用户会不会留下。  
> 特色能力决定用户为什么记住你。

AI 测试方法论必须尊重这个顺序。

## 三、把用户路径作为最高级测试对象

AI 写代码时，最容易陷入模块视角：

- Core 实现了吗？
- CLI 命令有了吗？
- GUI 调通了吗？
- 类型过了吗？

这些都重要，但它们不是用户的世界。

用户的世界是路径：

- 我今天打开邮箱，想看看有没有新邮件。
- 我收到一封重要邮件，想确认内容并回复。
- 我写了一半，临时离开，希望草稿别丢。
- 我发了一封带附件的邮件，希望知道它是否真的发出去了。
- 我看到一封付款相关邮件，希望知道它能不能信。
- 我网络断了，希望产品别把我带到一个混乱状态。

所以 AI 测试应该从“用户路径”开始，而不是从“技术模块”开始。

对每条用户路径，我们都要写出三个层次：

### 用户目标

用户真正想完成什么。

例如：

> 用户想发送一封带附件的邮件，并确信邮件已经发出。

### 产品承诺

产品在这条路径上必须保证什么。

例如：

> 如果发送成功，用户能看到明确结果；如果失败，草稿和附件不能丢，错误原因要可理解。

### 自动化证据

AI 如何证明这个承诺成立。

例如：

> CLI E2E 能发送带附件邮件，在已发送中读回，附件内容可比对；GUI 测试能看到发送中、成功、失败恢复状态。

这样测试就不再是孤立的技术检查，而是产品承诺的证据链。

## 四、CLI 是事实层，GUI 是体验层

SealMail 的新架构让这套方法论更容易落地。

因为现在业务能力集中到了 CLI：

```text
GUI -> cli_json bridge -> CLI JSON -> Core
AI 测试 -> CLI JSON -> Core
CI 测试 -> CLI JSON -> Core
真实邮箱 E2E -> CLI JSON -> Core
```

这让我们可以把测试分成两层。

### 1. CLI 事实层测试

CLI 测试回答：

> 事情实际上有没有发生？

比如：

- 邮件是否真的同步到了？
- 邮件是否真的发出去了？
- 附件是否真的存在？
- 删除是否真的进入回收站？
- 过滤器是否真的移动了邮件？
- 验签结果是否真的正确？
- 错误输出是否结构化？

CLI 的优势是稳定、可重复、可断言。

它非常适合 AI 自动化，因为 AI 不需要猜截图，也不需要通过视觉判断状态。

CLI JSON 是产品事实层。

### 2. GUI 体验层测试

GUI 测试回答：

> 用户完成这件事时，是否顺手、清楚、不累？

比如：

- 用户能否快速找到写信入口？
- 邮件列表是否清楚表达未读、星标、可信状态？
- 同步中是否有恰当反馈？
- 发送失败时，草稿是否还在界面里？
- 风险提示是否清楚但不过度打扰？
- 附件下载、预览、保存是否符合直觉？

GUI 测试不应该重复证明所有业务事实。

它应该验证体验层：

> 用户看到了什么、理解了什么、下一步是否明确。

所以新架构的价值不是“我们可以只测 CLI”。

真正的价值是：

> CLI 让事实可靠，GUI 测试让体验可靠。

这两层结合，才能服务产品哲学。

## 五、AI 测试要建立证据等级

不同测试提供的证据强度不同。

AI coding 最大的问题之一，是容易把弱证据当强证据。

比如：

- 类型通过，不代表用户路径顺。
- 单元测试通过，不代表真实邮箱可用。
- CLI 能发信，不代表 GUI 写信体验好。
- 截图看起来正常，不代表底层状态正确。

所以我们需要证据等级。

### L0：静态约束

检查代码结构有没有违反架构原则。

例如：

- GUI 业务 API 不允许直接 invoke 旧 Core command。
- 密钥不能出现在 git、日志、命令行参数里。
- 前端类型必须通过。

L0 的价值是防止明显偏航。

### L1：核心逻辑测试

验证纯业务逻辑。

例如：

- 邮件解析。
- 签名和验签。
- 风险识别。
- 过滤器匹配。

L1 的价值是让 Core 可信。

### L2：接口契约测试

验证 CLI JSON 合约。

例如：

- 命令是否存在。
- JSON 字段是否稳定。
- 错误是否可解析。
- secret 是否不会出现在输出中。

L2 的价值是让自动化入口可信。

### L3：本地集成测试

验证 Store、DB、缓存、草稿、联系人、过滤器等组合行为。

L3 的价值是证明本地产品状态可持续。

### L4：真实环境 E2E

验证真实邮箱、真实网络、真实协议。

例如：

- Gmail IMAP/SMTP。
- 真实发信。
- 真实目录移动。
- 真实附件读回。

L4 的价值是打破“模拟环境里一切正常”的幻觉。

### L5：GUI 体验回归

验证用户真的能通过界面完成任务。

例如：

- 首次添加账户。
- 收件箱同步和打开邮件。
- 写信、附件、草稿恢复。
- 风险提示出现时是否不遮挡核心操作。

L5 的价值是保护产品体验，而不只是保护功能。

一个严肃的 AI 交付，不能只说“测试过了”。

它应该说：

> 我拿到了哪些等级的证据？  
> 哪些用户路径有强证据？  
> 哪些还只有弱证据？  
> 哪些体验风险还没覆盖？

## 六、AI coding 的“完成”必须重新定义

过去的完成，常常是：

> 功能写完了，测试也过了。

这太粗糙。

面向产品哲学的完成，应该是：

> 用户路径被定义了。  
> 产品承诺被写清楚了。  
> 基本盘没有被特色功能伤害。  
> 关键异常场景有恢复策略。  
> 自动化证据覆盖了事实层和体验层。  
> 剩余风险被明确列出来，而不是被忽略。

这意味着 AI 在交付前必须做一次产品审计，而不是只做代码审计。

产品审计要问：

- 这次改动是否改善了用户日常高频路径？
- 有没有让某个常用操作变慢、变绕、变不清楚？
- 特色能力是否抢了核心体验的注意力？
- 错误状态是否更清楚、更可恢复？
- 用户是否需要理解更多内部概念才能完成同样任务？
- 我手里的测试证据，是否足以支撑“完成”这个说法？

只有这些问题都有答案，AI 才应该说“完成”。

## 七、把用户手测变成产品资产

用户每次手测发现问题，都是产品知识。

如果这些知识只停留在聊天里，就会反复丢失。

正确做法是把它们沉淀为产品资产：

- 用户路径文档。
- 验收契约。
- CLI 场景测试。
- GUI 体验测试。
- 异常场景清单。
- 回归检查表。

这样下一次 AI coding 不会从零开始。

它会继承项目已经学到的东西。

这也是 AI 测试方法论的核心价值之一：

> 让产品经验可以积累，让用户不必一次次替 AI 重新发现同样的问题。

## 八、SealMail 的方法论落点

针对 SealMail，我认为应该建立四类长期资产。

### 1. 产品哲学文档

说明 SealMail 首先是邮件客户端，其次才是可信邮件客户端。

它保护的是产品方向。

### 2. 用户路径文档

列出日常邮件基本盘：

- 首次添加账户。
- 同步收件箱。
- 阅读邮件。
- 写信和发信。
- 草稿恢复。
- 附件处理。
- 文件夹管理。
- 搜索和筛选。
- 退出重开。
- 网络和账号异常。

再列出特色能力路径：

- 可信邮件。
- 未知签名。
- 篡改邮件。
- 冒充邮件。
- 高风险邮件。
- 信任联系人管理。

### 3. 自动化证据矩阵

每条用户路径都要标明：

- CLI 是否覆盖？
- GUI 是否覆盖？
- 是否需要真实邮箱？
- 是否需要截图或交互验证？
- 当前证据等级是什么？

这样我们就不会把“CLI 通过”误认为“产品体验完成”。

### 4. AI 交付清单

每次 AI 改动交付前，都要回答：

- 改动影响了哪些用户路径？
- 核心体验有没有被破坏？
- 特色能力有没有过度打扰？
- 我跑了哪些证据等级？
- 哪些风险没有覆盖？
- 是否需要用户手测；如果需要，手测重点是什么？

这不是流程负担。

这是为了减少反复返工。

## 九、最终目标：让 AI 成为产品质量的放大器

AI coding 最大的价值，不应该只是写代码更快。

如果只是写得更快，也可能只是更快地制造返工。

真正的价值应该是：

> AI 能把产品哲学转化为稳定的验收标准，  
> 把用户路径转化为自动化剧本，  
> 把一次次手测经验转化为长期回归资产，  
> 把“我觉得完成了”变成“我有证据证明它更接近好产品”。

这才是 AI 测试方法论要解决的问题。

它不是为了让测试更多。

它是为了让产品更少偏离用户真正需要的东西。

对于 SealMail，这句话尤其重要：

> 不要让可信验证这个特色，掩盖邮件客户端的基本盘。  
> 也不要让邮件客户端的基本盘，失去可信验证带来的独特价值。

好的 AI 测试方法论，要同时守住这两件事。

它既要保护日常体验，也要保护产品特色。

它既要验证事实，也要验证体验。

它既要服务工程正确性，也要服务产品哲学。

这才是我们真正需要的测试方法论。
