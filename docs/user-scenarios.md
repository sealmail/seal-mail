# SealMail 用户场景与产品承诺

这份文档定义 SealMail 的核心用户路径。

它不是功能清单，而是 AI coding 和 AI testing 共同使用的产品验收地图。

每条场景都包含：

- 用户目标：用户真正想完成什么。
- 产品承诺：SealMail 必须让用户获得什么确定性。
- 事实证据：CLI/Core 层应该证明什么。
- 体验证据：GUI 层应该证明什么。
- AI 评审重点：大模型应该重点判断哪些产品哲学风险。

## 设计原则

SealMail 首先是邮件客户端，其次才是可信邮件客户端。

因此场景分成两组：

1. **基本盘场景**：收、看、写、发、同步、附件、草稿、归档、删除。
2. **特色能力场景**：签名、验证、可信联系人、风险提示。

基本盘场景优先级高于特色能力场景。

特色能力必须增强邮件体验，不能压过邮件体验。

## 证据等级

每条场景按以下证据等级逐步覆盖：

- L0 静态约束：架构和敏感信息不偏航。
- L1 Core 逻辑：纯业务逻辑正确。
- L2 CLI 契约：JSON 输入输出稳定、可自动化。
- L3 本地集成：Store、DB、缓存、草稿、联系人等状态正确。
- L4 真实环境：真实邮箱、真实协议、真实网络。
- L5 GUI 体验：用户通过界面能清楚、顺手地完成任务。

## 基本盘场景

### S01 首次添加邮箱账户

用户目标：

用户想把自己的邮箱加进 SealMail，并确认它能正常收发。

产品承诺：

- 用户知道自己正在配置哪个邮箱。
- 登录失败时，错误原因可理解。
- 密码或应用专用密码不出现在日志、URL、命令行参数或输出中。
- 成功后，账号出现在账户列表里。

事实证据：

- CLI `account test-json` 能用 stdin JSON 测试账户。
- CLI `account add-json` 能保存账户和 secret。
- `state --json` 不暴露 secret。
- `.env.local` 等敏感文件不被 git 跟踪。

体验证据：

- GUI 能清楚区分普通密码、应用专用密码、OAuth。
- 用户失败后知道下一步如何修复。
- 成功后能自然进入收件箱，而不是停在技术配置页。

AI 评审重点：

- 是否把普通用户当成安全专家？
- 错误文案是否说人话？
- 安全要求是否必要且不过度打扰？

### S02 日常打开收件箱

用户目标：

用户打开 App，想快速知道有没有新邮件，并进入日常处理状态。

产品承诺：

- 当前账户、当前文件夹、邮件列表状态清楚。
- 同步中、同步失败、缓存状态可理解。
- 邮件列表是主角，可信状态是辅助信息。

事实证据：

- CLI `folders` 能返回目录列表。
- CLI `sync` 能同步 INBOX。
- CLI `list` 能分页返回邮件列表和 total。

体验证据：

- GUI 首屏优先呈现邮件客户端基本结构。
- 用户能快速识别账号、目录、邮件列表。
- 安全提示不应该占据首屏主视觉，除非当前邮件确实高风险。

AI 评审重点：

- 这看起来首先像邮件客户端，还是像安全告警系统？
- 用户是否能不思考地开始处理邮件？
- 同步状态是否诚实且不过度紧张？

### S03 阅读邮件

用户目标：

用户想打开一封邮件，阅读正文、发件人、时间、附件和相关状态。

产品承诺：

- 正文阅读舒适。
- 发件人和主题清楚。
- 附件可见且可操作。
- 可信状态存在，但不破坏阅读。
- 高风险邮件需要明确提醒。

事实证据：

- CLI `read` 能返回完整邮件、正文、附件、trust、risk。
- CLI `thread` 能返回同会话邮件。

体验证据：

- GUI 的阅读区域有足够空间。
- 可信状态层级清楚：可信安静、未知温和、高风险明确。
- 技术细节默认不压过正文。

AI 评审重点：

- 邮件正文是否仍是主角？
- 风险提示是否有分寸？
- 用户是否知道下一步可以回复、归档、删除或保存附件？

### S04 写信并发送

用户目标：

用户想写一封普通工作邮件并发送。

产品承诺：

- 写信入口容易找到。
- 收件人、主题、正文、附件操作自然。
- 发送中和发送结果明确。
- 发送失败不丢草稿。
- 签名能力默认增强可信度，但不打扰写信。

事实证据：

- CLI `send` 能发送邮件。
- 发送结果返回 signed、method、fingerprint。
- 真实邮箱中能在已发送目录读回邮件。

体验证据：

- GUI 写信流程以“写邮件”为中心，而不是以“签名”为中心。
- 附件状态清楚。
- 发送失败时用户能继续编辑或重试。

AI 评审重点：

- 写信体验是否自然？
- 签名信息是否喧宾夺主？
- 失败状态是否诚实、可恢复？

### S05 草稿恢复

用户目标：

用户写了一半离开，希望回来还能继续。

产品承诺：

- 草稿自动保存或明确保存。
- 退出重开不丢。
- 发送成功后草稿状态清理合理。

事实证据：

- CLI `draft save`、`drafts`、`draft delete` 可 roundtrip。
- Store 重新加载后草稿仍存在。

体验证据：

- GUI 草稿状态可见但不打扰。
- 用户不会误以为草稿已经发送。

AI 评审重点：

- 草稿体验是否减少焦虑？
- 状态文案是否避免误导？

### S06 附件处理

用户目标：

用户想发送、查看、保存附件。

产品承诺：

- 附件是否已添加、上传或保存成功要清楚。
- 保存附件失败时原因可理解。
- 附件内容不能被静默损坏。

事实证据：

- CLI `send --attach` 能发送附件。
- CLI `read` 返回附件列表。
- CLI `attachment save` 保存后可做内容比对。

体验证据：

- GUI 附件控件清楚、可移除、可重试。
- 附件不应被安全状态区域遮挡。

AI 评审重点：

- 用户是否能相信附件真的被带上了？
- 失败时是否知道下一步？

### S07 移动、归档、删除、标记

用户目标：

用户想快速整理邮件。

产品承诺：

- 已读、星标、移动、归档、删除符合常见邮件客户端习惯。
- 删除和永久删除区分清楚。
- 操作失败不制造假成功。

事实证据：

- CLI `mark`、`flag`、`move`、`archive`、`delete` 可执行。
- 真实邮箱中移动和删除能被读回验证。

体验证据：

- GUI 操作入口稳定、反馈明确。
- 误操作有恢复空间，尤其是删除。

AI 评审重点：

- 整理邮件是否轻快？
- 安全特色是否影响普通整理动作？

## 特色能力场景

### S08 可信邮件

用户目标：

用户收到一封重要邮件，想知道它是否可信。

产品承诺：

- 可信邮件应该给用户安心感，但不打扰正常阅读。
- 可信依据可以展开查看。

事实证据：

- CLI `read` 返回 `trust=verified`。
- 签名、hash、fingerprint 验证正确。

体验证据：

- GUI 显示可信状态清楚但克制。
- 技术细节默认折叠。

AI 评审重点：

- 可信状态是否增强判断，而不是抢走注意力？

### S09 未知签名

用户目标：

用户收到一封带签名但未信任的邮件，想知道该如何处理。

产品承诺：

- 不把未知签名误导成可信。
- 不把未知签名渲染成严重危险。
- 给出合理下一步：查看详情、加入信任、谨慎处理。

事实证据：

- CLI `read` 返回 `trust=signedUnknown`。

体验证据：

- GUI 使用温和提示，不制造不必要焦虑。

AI 评审重点：

- 提示是否有分寸？
- 用户是否能理解“签名有效但未信任”的含义？

### S10 篡改或冒充

用户目标：

用户需要避免被错误邮件误导。

产品承诺：

- 篡改和冒充必须强提示。
- 风险语言必须清楚、可行动。
- 不让用户误以为邮件可信。

事实证据：

- Core 测试覆盖 tampered、impersonation。
- CLI `read` 返回对应 trust/risk。

体验证据：

- GUI 明确阻止或强提醒高风险动作。
- 提示说明影响和建议，而不是只给技术术语。

AI 评审重点：

- 是否在关键风险上足够坚定？
- 是否避免恐吓式表达？

### S11 高风险语义邮件

用户目标：

用户收到付款、登录、合约、转账相关邮件，想知道是否需要谨慎。

产品承诺：

- 风险提示要帮助用户判断。
- 不要把每封普通邮件都变成风险事件。
- 用户能继续完成正常邮件操作。

事实证据：

- Core 风险识别测试覆盖 fund、account、contract。

体验证据：

- GUI 风险提示层级合理。
- 提示内容可行动。

AI 评审重点：

- 特色能力是否在正确时机出现？
- 是否压过正常阅读和回复？

## 全局反模式

任何场景中都应避免：

- 让安全特色压过邮件客户端基本盘。
- 使用用户不理解的内部术语。
- 把失败包装成成功。
- 静默丢失草稿、附件、移动或删除结果。
- 用默认值掩盖本不该为空的状态。
- 让用户为了完成普通操作理解 Core、CLI、签名、hash、fingerprint 等内部机制。

## AI 评审的核心问题

每次 AI evaluator 都应该围绕这些问题判断：

1. 这个路径是否首先把用户的核心任务做好？
2. 特色能力是否增强而不是干扰？
3. 状态、结果和错误是否诚实？
4. 用户是否知道下一步怎么做？
5. 产品是否减少了用户的负担，而不是增加负担？
